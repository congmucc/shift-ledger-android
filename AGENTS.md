# AGENTS.md — Shift Ledger Android 工程规则

## 第一性原理
- 先还原真实目标、约束、成功标准，再改代码。
- 用户口述里的例子只当线索，不机械照抄；实现应服务工时账本的长期一致性。
- 只做当前目标必需的架构整理，不顺手重构无关模块。

## 优先复用库与官方模式
- 能用 Flutter 官方组件、官方推荐插件或成熟库解决的，不手搓底层交互。
- 日期、时间、文件保存、日历、选择器等通用能力优先采用官方/成熟库，然后在项目内做薄封装。
- 手写代码只负责业务语义、展示模型、状态编排和项目统一风格，不重复造系统控件。
- 引入依赖前检查：维护状态、平台支持、是否符合 Android/iOS 官方交互、是否显著减少自研代码。

## 架构复用
- 日期/时间、年月、发薪日、班次类型、计薪规则等常见选择能力必须沉到共享组件或工具层。
- 页面不得各自实现一套日期格式化、时间选择、日历标记规则。
- 共享封装要薄：统一入口、统一文案、统一无障碍语义，但不要为了抽象制造复杂框架。

## UI 与可访问性
- 大字号是适配目标，不是单独“大字号模式”。系统字体调大时，关键日期、工时、按钮和操作不能遮挡或消失。
- 日历和汇总不能只靠颜色表达信息；备注、夜班、加班、多段等状态必须有文字/图标/语义标记。
- 老年用户优先：触控目标不小于 44dp，文案直白，选择器优先使用系统熟悉的日历/滚轮/底部弹层。
- 卡片密度可调整，但不能牺牲核心数字可读性。

## Android 文件保存
- CSV/JSON 等用户要拿走的文件优先走系统保存/分享/文件选择器能力，不硬写公共 Downloads 路径作为唯一方案。
- 只有在确需直接访问共享存储时才申请权限，并且必须处理拒绝、取消和不同 Android 版本差异。
- 错误提示要说明真实原因，不用“预览环境不支持”掩盖权限、取消或系统策略问题。

## 验证
- 行为变更先补测试或最小复现，再实现。
- 完成前至少运行 `flutter test`；涉及依赖或语法结构时同时运行 `flutter analyze`。
