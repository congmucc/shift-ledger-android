# Shift Ledger Design System / 工时账本设计基线

## 设计定位

Shift Ledger 是一本私人“工时账本”。它不是企业考勤后台，也不是排班审批系统。设计目标是让用户快速确认：

- 今天有没有记完整；
- 哪天上了哪些班、几段、多久；
- 加班、夜班、补贴、扣款和收入估算是否可信；
- 数据是否能导出、备份和恢复。

整体气质：**暖色纸面、安静账本感、日历优先、数字可信、低压私密**。

## 当前信息架构

底部导航采用五个视觉位置：

1. **首页**：今日记录、本周期进度、快捷操作。
2. **日历**：月视图/列表视图、今日跳转、日期详情、补一段。
3. **中间新增**：主操作入口，打开新增/编辑工时记录 sheet。
4. **汇总**：本月/本周/年度/发薪周期/自定义统计和 CSV 导出。
5. **设置**：班次模板、计薪规则、规则历史、加班规则、夜班规则、发薪周期、导出与备份、WebDAV。

## 视觉原则

### 1. 账本优先，不做后台感

- 背景使用暖纸面色，不使用冷蓝白企业后台风。
- 卡片边框轻、阴影少，像纸面分组而不是管理系统面板。
- 数字和日期是主角，装饰只能辅助理解。

### 2. 数字可信

- 工时、收入、天数、次数使用强字重和稳定数字排版。
- 汇总数字必须能下钻到具体日期或记录。
- CSV 字段和页面展示的统计口径必须一致。

### 3. 日历优先

- 日历是独立主页面，不是首页里的附属组件。
- 月视图必须同时表达有工时、加班、夜班、备注、今日，不能只靠单一颜色覆盖。
- 列表视图用于补足月网格不适合承载的文字信息。

### 4. 老年用户与大字号优先

- 触控目标不小于 44dp。
- 系统字体变大时，关键日期、工时、按钮和底部导航不能消失。
- 可读性优先于信息密度；必要时局部限制文字缩放，但不能隐藏关键语义。

### 5. 私密和可控

- 备份、恢复、WebDAV、自动备份都要明确告诉用户会发生什么。
- 敏感密码不进入普通备份。
- 恢复前必须强调覆盖风险。

## 字体与排版

当前实现使用系统字体和 Material 3 text theme：

- 页面标题：28sp，粗字重，短标题优先。
- Section 标题：22sp，粗字重。
- 卡片核心数字：24-34sp，粗字重，tabular figures。
- 正文：15sp，行高约 1.45。
- 辅助标签：13sp，muted 色。
- 底部导航：12sp，粗字重，最大缩放受控，避免挤出布局。

## 组件语义

### PageFrame

- 页面统一容器。
- 顶部标题 + 可选右侧动作。
- 底部留出导航栏安全空间。

### LedgerCard

- 默认卡片组件。
- 24dp 圆角、暖白底、浅边框、无重阴影。
- 用于首页指标、日期详情、设置分组、汇总区块。

### MetricCard

- 用于核心数字。
- 首页“今日已记录”和汇总核心指标应优先使用。
- 可点击时表示可以进入详情或编辑。

### WorkEntryTile

- 展示单段工时。
- 必须包含时间范围、类型、净工时、地点/岗位、补贴/扣款/备注等摘要。
- 编辑按钮触控目标不小于 44dp。

### Calendar Cell

- 使用 `table_calendar` 承载网格。
- 每个日期需要语义化描述：日期、工时、段数、是否加班/夜班/备注/今日。
- 视觉标记使用多点/标签组合，不以单一颜色作为唯一信息来源。

### Bottom Navigation

- 视觉上有 5 个位置，中间“＋”是主操作，不对应普通页面。
- 首页、日历、汇总、设置是页面入口。
- 底部栏保持大触控区域和明确选中状态。

## 表单与弹层

- 新增/编辑工时记录使用底部 sheet。
- 单段编辑使用对话框，包含开始/结束、休息分钟、类型、计薪规则、地点/岗位、补贴、扣款、备注。
- 日期、时间、月日选择使用共享 picker 封装，不允许页面各写一套。
- 删除当天记录、恢复备份、WebDAV 恢复等破坏性操作必须二次确认。

## 导出与备份体验

- CSV 和 JSON 导出必须走系统保存面板或明确的可替代保存入口。
- “取消保存”和“保存失败”要区分。
- 本地恢复只能从 App 已保存的私有备份副本恢复。
- WebDAV 密码缺失、配置不完整、上传失败、等待限流、内容未变化跳过，都要有用户可理解的状态文案。

## 可访问性验收

至少覆盖：

- 首页、日历、汇总、设置在普通字号下主要内容不被底部导航遮挡。
- 日历在 2.0 text scale 下不抛布局异常。
- 日历日期 cell 不只靠颜色表达状态。
- 重要按钮有文字或 tooltip/semantics。
- 删除、恢复、覆盖类动作有确认弹窗。

## 当前明确不追求

- 不做企业后台式高密度表格。
- 不做复杂 BI 图表。
- 不做炫酷动效或品牌插画优先。
- 不为未来多端同步预埋复杂 UI。
- 不用设计系统抽象替代真实业务语义。
